02
The problem
What the baseline left on the table

20/20 evals.
Still not deployable.

The baseline agent passed every generic eval but the answers it produced weren't customer-ready. Four failure modes were hiding underneath the green checkmarks.

✕
JSON parse failures
Complex questions caused the model to emit reasoning text. Fallback path kicked in and the customer-facing field carried raw chain-of-thought.
✕
Ungrounded numbers
Plausible figures appeared in answers that no cited source could support. LLM-as-judge missed this systematically.
✕
Toy retrieval
Set-intersection keyword overlap. Worked on questions phrased like the KB; broke on natural-language paraphrases.
✕
No critique loop
Single-shot drafting. No mechanism to catch a weak answer before sending it through to the customer.
03
Architecture
Five stages, one pipeline, per question, in parallel

From question to grounded answer.

01 · Retrieve
BM25
Token-level ranking over content + tags. Returns top-K candidates from the KB.
02 · Rerank
Claude Haiku
Scores each candidate 0–100 for question relevance. Combines with BM25 for the final order.
03 · Draft
Sonnet · forced tool
Structured output via tool_choice. Zero parse failures possible.
04 · Critique
Sonnet · 5 criteria
grounded · cited_correctly · confidence_calibrated · tone · addresses_question. Fail → revise.
05 · Verify
Programmatic
Every numeric claim must appear in a cited source. Zero-cost ground-truth check.

One cross-answer reviewer runs at the end to catch contradictions across the batch.

04
Innovation · 01
Retrieval that survives paraphrase

BM25 + Claude rerank.

Baseline
Set-intersection keyword overlap
Counted shared tokens between query and (content + tags). Brittle on natural-language phrasing, blind to relevance gradients.
Pro
BM25 over (content + tags) → top-K → Claude scores 0–100 → final blend 30/70
Token-level frequency-aware ranking plus an LLM relevance judge over the top-K. ~1 cheap Haiku call per question for rerank. Surfaces the right document even when the question and the source use different vocabulary.
05
Innovation · 02
No more "model returned text we couldn't parse"

Structured output, by construction.

Baseline
"Return JSON" in the system prompt
When the model emitted chain-of-thought, the fallback put raw reasoning into the customer-facing field. Surprise RFP S2 and S3 failed this way.
Pro
tool_choice: {type: "tool", name: "submit_answer"}
A typed tool schema with required fields including evidence_quotes. The model MUST call the tool. The API guarantees the input matches the schema. There is no parse step — therefore no parse failure.
06
Innovation · 03
Reflexion — the agent reviews its own work

Draft → critique → revise.

This run · 0 revisions needed
Every draft passed all five criteria first time
grounded · cited_correctly · confidence_calibrated · tone_professional · addresses_question
Critic is on standby
Catches the failure mode the moment it appears
When grounding fails, the critic flags should_revise=true and writes specific notes for the revise pass.
07
Innovation · 04
The check LLM critics systematically miss

Citation verifier.

Caught in S2
Ungrounded numeric claims
$33, $29.70, $891,000, 2,500

The verifier extracted these from the answer text and could not find them in any cited source. The runner automatically downgraded confidence + added a human-review flag.
How it works
Pure programmatic check
Regex over numeric / currency / percent / certification claims, normalized against the joined content of every cited source. Zero false negatives on string-grounded claims. No model in the loop = cheap + deterministic.
